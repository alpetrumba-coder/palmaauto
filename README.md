# ПальмаАвто — веб-проект

**Основная папка проекта:** `c:\cursor\palmaauto` — весь код и документация только здесь.

Раньше проект лежал на медленном сетевом диске; актуальная копия для разработки — **на локальном диске C**. Если у вас был файл `.env` только в старой папке, скопируйте его сюда вручную.

## Что уже сделано

**Этап 1**

- **Next.js** (App Router) + **TypeScript** + **ESLint**
- **Prisma** + **PostgreSQL**: модель `User` и стартовая миграция
- Глобальные стили в духе минималистичных интерфейсов (светлая/тёмная тема по системным настройкам)
- Публичный layout для витрины; админка — отдельно **`/admin-panel`**

**Этап 2**

- Модели **`Car`**, **`CarImage`**, миграция `20250410120000_add_cars_and_images`
- Публичные страницы **`/cars`** и **`/cars/[slug]`** (только `active = true`)
- Сид демо-данных: **`npm run db:seed`** (после миграций; фото — локальные файлы в **`public/cars/...`**, см. ниже)

Подробная карта папок: **[docs/STRUCTURE.md](./docs/STRUCTURE.md)**.

### Фото автомобилей (рекомендуемый простой вариант)

1. Положите файлы в **`public/cars/<slug>/`**, где **`<slug>`** совпадает с идентификатором машины в админке (например `toyota-camry`, `1.jpg`).
2. В админ-панели (**`/admin-panel/cars`**) в поле каждого фото укажите путь **как на сайте**: **`/cars/toyota-camry/1.jpg`** (с ведущим слэшем).
3. Подробная памятка лежит в **`public/cars/README.txt`**.
4. Если вместо файлов используете внешний URL **`https://...`**, добавьте домен в **`next.config.ts`** → **`images.remotePatterns`**.

## Перед командами `npm`: установлен ли Node.js?

Если PowerShell пишет **`npm : Имя "npm" не распознано`** (как в вашем терминале), значит **Node.js не установлен** или папка с ним **не в переменной PATH**.

1. Установите **LTS** с официального сайта: [https://nodejs.org/](https://nodejs.org/)  
   В установщике оставьте опцию **Add to PATH** / **Добавить в PATH**.
2. Закройте **все** окна терминала и перезапустите **Cursor** (или хотя бы новый терминал), чтобы подтянулся обновлённый PATH.
3. Проверка:

   ```powershell
   node -v
   npm -v
   ```

   Должны отобразиться номера версий без ошибки.

Если Node вы уже ставили, но команд всё равно нет: откройте `C:\Program Files\nodejs\` в проводнике — там должны быть `node.exe` и `npm.cmd`. Добавьте **эту папку** в системную переменную **Path** (Параметры → Система → О системе → Дополнительные параметры системы → Переменные среды).

## Быстрый старт

1. Установите зависимости (из `s:\cursor\palmaauto`):

   ```bash
   npm install
   ```

2. Создайте файл **`.env`** (скопируйте из `.env.example`) и укажите рабочий `DATABASE_URL` для PostgreSQL.

3. Примените миграции к базе:

   ```bash
   npx prisma migrate dev
   ```

   Затем (по желанию) демо-машины в каталог:

   ```bash
   npm run db:seed
   ```

   Без рабочей базы страницы с данными из БД (каталог `/cars`) при запросе вернут ошибку.

4. Клиент Prisma после `npm install` создаётся автоматически (`postinstall` → `prisma generate`). При необходимости вручную:

   ```bash
   npm run db:generate
   ```

5. Запуск в режиме разработки:

   ```bash
   npm run dev
   ```

   Откройте в браузере [http://localhost:3000](http://localhost:3000).

6. Проверка production-сборки:

   ```bash
   npm run build
   ```

   Скрипт при **заданном** `DATABASE_URL` выполняет `prisma migrate deploy`, затем `next build`. Без переменной миграции пропускаются (удобно для сборки без локальной БД). Только Next.js: `npm run build:next`.

## Деплой на Vercel

1. В **Settings → Environment Variables** задайте **`DATABASE_URL`** — строка PostgreSQL (у Neon обычно **Pooled**; `sslmode=require` уже в строке). Приложение само добавит **`pgbouncer=true`** для pooler-хоста (иначе Prisma на serverless падает).
2. Опционально **`DIRECT_URL`** — строка **Direct** из Neon (без `-pooler` в хосте): надёжнее **`prisma migrate deploy`** на сборке. Если не задана, миграции идут через ту же **`DATABASE_URL`**, только без режима pgbouncer.
3. Команда сборки — **`npm run build`**: при заданном **`DATABASE_URL`** выполняется **`prisma migrate deploy`**, затем **`next build`**.
4. Чтобы в каталоге были машины, один раз выполните сид локально против той же БД (`npm run db:seed`) или добавьте записи через Prisma Studio.

## Авторизация (этап A)

- В **`.env`** задайте **`AUTH_SECRET`** (случайная строка, например `openssl rand -base64 32`). На **Vercel** добавьте ту же переменную.
- **Регистрация и вход:** `/register`, `/login`. Сброс пароля: **`RESEND_API_KEY`**, **`EMAIL_FROM`** (см. `.env.example`), опционально **`AUTH_URL`** с публичным URL сайта для ссылок в письмах.
- **Первый администратор:** в **`.env`** задайте **`INITIAL_ADMIN_EMAIL`** и **`INITIAL_ADMIN_PASSWORD`**, затем выполните **`npm run db:seed`** — пользователь с ролью ADMIN будет создан или обновлён.
## Бронирование (этап B)

- На странице автомобиля **`/cars/[slug]`** можно выбрать даты и создать бронь (нужен вход). Пересечения с бронями в статусах **ожидает оплаты** и **оплачено** не допускаются.
- **`/account`** — список своих броней; отмена доступна для статуса «ожидает оплаты». Оплата — следующий этап.

### Админ-панель (отдельный вход)

- В подвале ссылка **«Админпанель»** → **`/admin-panel/login`**. Это **не** обычный вход на сайт: используйте **`INITIAL_ADMIN_EMAIL`** и **`INITIAL_ADMIN_PASSWORD`** из **`.env`** (те же, что для **`npm run db:seed`**). Нужен **`AUTH_SECRET`** для cookie.
- После входа: **`/admin-panel`** → раздел **«Автомобили в прокате»** (**`/admin-panel/cars`**) — создание, редактирование и удаление машин и списка фото (в БД хранятся пути **`/cars/...`** или внешние URL). Новые файлы кладите в **`public/cars/<slug>/`** на диске проекта или на сервере рядом со сборкой, затем укажите путь в форме (см. раздел «Фото автомобилей» выше).

После обновления схемы выполните **`npx prisma migrate deploy`** (или деплой на Vercel с `DATABASE_URL`).

## Проверка этапа 1 (по плану)

- **`npm run build`** — проект собирается без ошибок при настроенном `DATABASE_URL` (миграции применяются автоматически). Без БД: **`npm run build:next`**.
- **`npm run lint`** — линтер проходит.
- **Десктоп и телефон:** запустите `npm run dev`, откройте главную в браузере на широком экране и сузьте окно / инструменты адаптивности. Для реального телефона в той же Wi‑Fi сети:

  ```bash
  npm run dev:lan
  ```

  На телефоне откройте `http://<IP-вашего-ПК>:3000` (брандмауэр Windows может запросить разрешение для Node). Визуально те же токены (цвета, скругления, типографика), шапка и плейсхолдер каталога читаемы на узкой ширине.

## Полезные команды

| Команда | Назначение |
|--------|------------|
| `npm run dev:lan` | Dev-сервер на всех интерфейсах — удобно с телефона |
| `npm run lint` | Проверка ESLint |
| `npm run db:studio` | Prisma Studio — просмотр данных в БД |
| `npm run db:seed` | Заполнить каталог демо-авто (нужен применённый `migrate`) |
| `npm run db:push` | Упрощённая синхронизация схемы без файла миграции (только для прототипа) |

## Где искать пояснения в коде

- В начале многих файлов — комментарии **зачем** этот файл.
- В `components/README.md` и `lib/README.md` — что класть в эти папки.
- В `prisma/schema.prisma` — комментарии к моделям и enum.

Дальнейшие этапы плана (каталог, регистрация, бронирование, админка) расширяют этот каркас без смены корневой идеи структуры.
